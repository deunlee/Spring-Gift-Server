# TIL (Today I Learned)
오늘의 git 커밋에서 학습 가치가 있는 내용을 추출하여 `TIL.md`에 정리한다.

## 실행 절차

### 1단계: 오늘의 커밋 로그 수집
아래 명령어로 오늘 커밋 목록을 확인한다:
- `git log --since="$(date +%Y-%m-%d) 00:00:00" --oneline`

커밋이 하나도 없으면 "오늘 커밋이 없습니다"를 출력하고 **즉시 종료**한다.

### 2단계: 학습 가치 있는 커밋 필터링
커밋 메시지의 type을 기준으로 필터링한다.

**제외** (학습 가치 낮음):
- `chore` — 단순 유지보수
- `style` — 포맷팅, 코드 스타일
- `ci` — CI 설정

**포함** (학습 가치 높음):
- `feat` — 새로운 기능
- `fix` — 버그 수정
- `refactor` — 리팩토링
- `build` — 빌드/의존성
- `perf` — 성능 개선
- `docs` — 문서 (기술 내용이 포함된 경우)
- `test` — 테스트 작성
- 기타 type 없는 커밋도 내용을 확인하여 판단

필터링 후 학습 가치 있는 커밋이 없으면 "오늘 커밋 중 TIL로 정리할 내용이 없습니다"를 출력하고 **즉시 종료**한다.

### 3단계: 변경 내용 분석
선별된 각 커밋에 대해:
1. `git show <hash>`로 diff를 읽는다
2. 해당 커밋에서 배울 수 있는 **개념, 패턴, 기술**을 추출한다
3. 여러 커밋이 같은 주제를 다루면 하나의 TIL 항목으로 통합한다

### 4단계: 기존 TIL.md 확인 및 중복 제거
1. `TIL.md` 파일을 읽는다 (없으면 새로 생성할 준비)
2. 기존에 이미 다룬 주제와 **중복되는 내용은 제외**한다
3. 중복 판단 기준: 같은 개념/기술을 설명하는 경우 (제목이 다르더라도 내용이 겹치면 제외)

### 5단계: TIL 항목 작성 및 사용자 확인
추가할 TIL 항목을 아래 형식으로 **먼저 사용자에게 보여주고 승인을 받는다**.

```
📝 TIL 추가 항목 (N개)

### 항목 제목 1
개념 설명 텍스트...

---
### 항목 제목 2
개념 설명 텍스트...
```

**반드시 사용자 승인을 받은 후에만 TIL.md에 작성한다.**

### 6단계: TIL.md에 추가
승인을 받으면 `TIL.md` 파일의 **마지막에 내용을 추가**(append)한다.

## TIL 항목 작성 규칙

### 형식
- 각 항목은 `### 제목`으로 시작한다
- 제목 아래에 개념 설명 텍스트를 작성한다
- 항목 사이에 빈 줄 + `---` + 빈 줄로 구분한다
- 마지막 항목 뒤에도 빈 줄을 남긴다

### 내용
- **코드 스니펫 없이** 개념과 원리를 설명한다
- 단순 "무엇을 했다"가 아니라 **"왜 그렇게 하는지", "어떤 원리인지"**를 중심으로 작성한다
- 한 항목당 2~5문장으로 간결하게 작성한다
- 프로젝트 특화 내용보다 **범용적으로 재사용 가능한 지식**을 우선한다

### 예시
```markdown
### JPA 연관관계에서 orphanRemoval의 역할
부모 엔티티의 컬렉션에서 자식을 제거하면 자식 레코드도 DB에서 삭제되도록 하는 옵션이다.
`cascade = REMOVE`는 부모 삭제 시에만 동작하지만, `orphanRemoval = true`는 컬렉션에서 빠지는 것만으로도 삭제가 트리거된다.
고아 객체가 다른 곳에서 참조되지 않는 경우에만 사용해야 한다.
```

## 주의사항
- TIL.md가 존재하지 않으면 `# TIL` 헤더와 함께 새로 생성한다.
- 기존 내용은 절대 수정하거나 삭제하지 않는다. 항상 **끝에 추가**만 한다.
- 사용자 승인 없이 파일을 수정하지 않는다.
